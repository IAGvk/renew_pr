# ✅ **1. Problem Statement (What exactly are we solving?)**

### **Macro Context**

Large buyers (cloud, AI, industrials, utilities) are moving from *annual REC matching* to *hourly, locationally precise* clean‑energy consumption. This shift is not theoretical—real markets, standards, and tools are emerging **right now**:

*   **Google & Microsoft** are backing a marketplace for **time‑based energy certificates (Granular Certificates/GCs)** via LevelTen, designed to support 24/7 energy procurement. [\[datacenter...namics.com\]](https://www.datacenterdynamics.com/en/news/google-and-microsoft-back-247-carbon-free-energy-marketplace/)
*   **EnergyTag** has accredited GC issuers (e.g., **FlexiDAO**, **Energinet**, **Granular Energy**) enabling verified hourly certificates across 40+ countries. [\[energytag.org\]](https://energytag.org/energytag-accredits-first-granular-certificate-issuers-marking-a-major-milestone-for-hourly-clean-energy-tracking/), [\[granular-energy.com\]](https://www.granular-energy.com/insights/granular-energy-becomes-accrediated-issuer-of-granular-certificates-under-energytag)
*   **Google** is already converting global PPA portfolios into hourly certificates at scale using FlexiDAO’s system. [\[datacenter...namics.com\]](https://www.datacenterdynamics.com/en/news/google-to-expand-247-clean-energy-matching-with-flexidao/)
*   Buyers struggle because **hourly load rarely aligns with renewable generation**, creating **"residual carbon hours"** that remain unaddressed by traditional RECs. [\[energytag.org\]](https://energytag.org/projects/matching-for-googles-24-7-goal/)

### **Core Pain**

Today, even sophisticated sustainability teams cannot easily answer:

> “For every single hour of my operations in region X, was I using clean energy? If not, what is the cheapest, most reliable way to fix each dirty hour?”

The problem is messy because it requires **massive, fragmented data stitching**:

*   Hourly consumption by region/site
*   Hourly renewable production from PPAs or on‑site assets
*   Hourly grid emission factors
*   Hourly GC availability/pricing
*   Market rules per ISO/region
*   Portfolios spanning multiple geographies, certificates, suppliers

**No single tool today takes the user from raw data → insights → procurement → verification**, end‑to‑end.

### **Therefore:**

We build a **24/7 CFE Copilot**—an intelligent agent that:

1.  Ingests all consumption + renewable supply + emissions + certificate availability data
2.  Identifies every “dirty hour”
3.  Proposes mitigation options (GC purchases, storage, PPAs, load shifting)
4.  Executes procurement of **Granular Certificates** or generates a procurement‑grade recommendation
5.  Outputs a fully auditable 24/7 CFE compliance/reporting package

The copilot becomes both the **advisor** and the **execution engine** behind 24/7 CFE strategy.

# ✅ **2. Expected Positive Outcomes (Best Case vs. Worst Case)**

1.  **>95% hourly clean‑energy matching** achieved across global operations, something only top hyperscalers attempt today.
2.  **30–60% reduction in portfolio carbon intensity**, because the agent can match high‑carbon hours with low‑cost GCs globally.
3.  **20–40% savings** vs. consultant‑led procurement due to automated optimization + multi‑market shopping.
4.  **No regulatory surprises**: the system stays ahead of emerging hourly matching requirements (EU CBAM hourly delivery rules, GHG Protocol updates, SBTi revisions).
    *   Hourly matching is already entering regulation (EU CBAM requires proof that clean electricity is delivered in the *same hour* to avoid carbon taxes). [\[linkedin.com\]](https://www.linkedin.com/pulse/energytag-winter-wrap-up-powering-2026-energy-tag-lz07c)
5.  The copilot becomes the **default “CFE brain”** for procurement teams (like what Plaid did for bank data or Vercel did for web infra).

### **For the product/company**

1.  **High willingness to pay** (>$50k–$500k ARR per customer for enterprises).
2.  Early beachhead in a new but fast‑growing category validated by leaders like Google/Microsoft. [\[datacenter...namics.com\]](https://www.datacenterdynamics.com/en/news/google-and-microsoft-back-247-carbon-free-energy-marketplace/)
3.  Strong defensibility via:
    *   multi‑market data integrations
    *   GC marketplace connectors
    *   agentic optimization + explainability workflows
4.  Potential to expand into **24/7 energy procurement**, **PPA structuring**, **storage optimization**, **grid services**.

## **🌦️ Worst‑Case Positive Outcomes (Even the downside looks OK)**

### **For the user / enterprise**

1.  Users only achieve **60–70% hourly matching** because data gaps or procurement gaps limit optimization.
2.  Saves only **10–15%** in procurement cost vs. manual methods—but still reduces workload significantly.
3.  The copilot becomes a **reporting-only tool**, not a procurement engine, due to internal policy/legal constraints.

### **For the product/company**

1.  Becomes a **niche analytics dashboard**, not an autonomous agent.
2.  Targets only sustainability teams (smaller TAM) instead of procurement/finance.
3.  Growth limited by slow GC market adoption; however:
    *   GC accreditation is expanding (\~47+ countries via FlexiDAO; worldwide coverage via Granular Energy).     *   Global hyperscalers pushing 24/7 CFE guarantees long-term relevance.

Even in the worst case, you still have:

*   monthly recurring usage,

### Deliverables:

*   Data dictionary (consumption, production, emissions, certificates)
*   Success metric for pilot (e.g., "Close 80% of hourly gaps for a single region")

***

# **PHASE 2 (Weeks 2–4): Data Layer + Interoperability**

### Build the backbone:

1.  **Ingest**
    *   Hourly consumption from meters/utility APIs
    *   Hourly renewable generation data
    *   Hourly emissions (via Electricity Maps / public datasets)
    *   GC availability metadata from EnergyTag‑compatible issuers
        *   Energinet (Configuration 1) [\[energytag.org\]](https://energytag.org/energytag-accredits-first-granular-certificate-issuers-marking-a-major-milestone-for-hourly-clean-energy-tracking/)
        *   FlexiDAO (Configuration 3, 47+ countries) [\[energytag.org\]](https://energytag.org/energytag-accredits-first-granular-certificate-issuers-marking-a-major-milestone-for-hourly-clean-energy-tracking/), [\[datacenter...namics.com\]](https://www.datacenterdynamics.com/en/news/google-to-expand-247-clean-energy-matching-with-flexidao/)
        *   Granular Energy (global accreditation) [\[granular-energy.com\]](https://www.granular-energy.com/insights/global-energytag-accreditation?category=news)

2.  **Data model (Postgres)**
    *   `hourly_consumption`
    *   `hourly_supply`
    *   `emissions_factors`
    *   `certificates_inventory`
    *   `hourly_match_results`

3.  **Core capability**:
    *   Compute **hourly gap matrix** = consumption – renewable supply
    *   Compute carbon intensity per gap hour

### Deliverables:

*   Working data pipeline (ETL)
*   API to retrieve gaps by site/region
*   Preliminary dashboard with dirty hours highlighted

***

# **PHASE 3 (Weeks 4–6): Agentic Engine (Brain of the Copilot)**

### Build a LangGraph‑style multi‑agent system:

1.  **Gap‑Finder Agent**
    *   Identifies hours needing coverage
    *   Rank hours by carbon intensity (greedy heuristic)

2.  **Sourcing Agent**
    *   Knows GC availability across time/region
    *   Suggests best GC/portfolio fix
    *   Applies EnergyTag rules for hourly certificate validity
        *   EnergyTag’s standard defines how GCs must be timestamped & validated. [\[energytag.org\]](https://energytag.org/energytag-accredits-first-granular-certificate-issuers-marking-a-major-milestone-for-hourly-clean-energy-tracking/)

3.  **Feasibility Agent**
    *   Enforces location matching, time windows, and legal/contractual rules
    *   Ensures no double counting (critical for compliance)

4.  **Optimization Agent**
    *   Minimizes \$$ cost
    *   Minimizes ton‑hours of carbon
    *   Maximizes certificate authenticity

### Deliverables:

*   Full agentic flow from “dirty hours → recommendations”
*   Python optimizer (ILP or heuristic)
*   Test suite for 10–20 scenarios

***

# **PHASE 4 (Weeks 6–8): User Interface + Explainability**

### Build dashboards that answer:

*   “Where are my hourly mismatches?”
*   “Which fixes are cheapest?”
*   “What is my projected % CFE coverage if we buy X GCs?”

### Explainability:

Every agent decision must output a **human‑readable narrative**, since CFE officers need audit‑ready reasoning.

### Deliverables:

*   React dashboard (site listing → hour heatmap → recommendations)
*   PDF/Excel export for ESG reports
*   API endpoints for enterprise workflows

***

# **PHASE 5 (Weeks 8–12): Pilot Deployment + Hardening**

### Pilot with 1–3 friendly customers:

1.  Validate ingestion pipelines
2.  Validate the hourly matching logic against their own spreadsheets
3.  Adjust optimization constraints

### Hardening:

*   Security baseline (authN/authZ)
*   Rate-limiting for agent actions
*   Monitoring for data freshness

### Deliverables:

*   Working pilot with \~80–90% functional coverage
*   Case study with measured benefits (# dirty hours mitigated, \$$ savings)
*   Roadmap for advanced features:
    *   GC transaction automation
    *   PPA structuring
    *   Multi‑market optimization
    *   Storage dispatch simulation
